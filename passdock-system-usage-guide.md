# PassDock 系统使用教程

## 这份文档是给谁看的

如果你现在的目标是：

- 把系统跑起来
- 知道后台每个页面是干什么的
- 按正确顺序把商品、支付、履约、交付配通
- 知道用户实际怎么买、运营实际怎么处理订单

那你优先看这份文档就够了。

这份文档不展开讲所有字段细节，而是讲“这个系统到底怎么用”。

如果你后面需要看字段级解释，再去看：

- [后台系统配置说明](./passdock-admin-system-config-guide.md)
- [配置说明](./passdock-config-reference.md)
- [接口草案](./passdock-api-draft.md)

## 先记住这个系统的主流程

当前系统推荐使用的是匿名下单模式：

1. 用户在前台选商品
2. 用户创建订单
3. 系统返回：
   - `订单号`
   - `order_access_token`
   - 用户侧可理解为“查单码”
4. 用户付款
5. 用户后续通过 `订单号 + 查单码`：
   - 查订单
   - 看交付结果
   - 上传支付凭证
   - 提交售后工单
6. 后台审核支付或等待自动确认
7. 系统履约发码
8. 系统站内或 Telegram 交付

一句话理解：

`PassDock = 商品售卖站 + 支付处理台 + 履约发码台 + 交付通知台`

## 首次上线时应该按什么顺序配置

第一次使用时，不要一上来就先建商品。

正确顺序是：

1. 先启动系统并登录后台
2. 配支付通道
3. 配 Provider
4. 配 Action
5. 配履约策略
6. 配交付策略
7. 再回到商品中心建商品和价格模板
8. 最后做一笔真实或半真实测试订单

原因很简单：

- 商品只是“卖什么”
- 支付通道决定“怎么收钱”
- 履约策略决定“怎么拿结果”
- 交付策略决定“怎么把结果发出去”

如果底座没配好，商品虽然能展示，但下单后不会真正闭环。

## 第 1 步：先把系统跑起来

你至少需要确认这几件事：

- 前端能打开
- 后端能启动
- 后台能登录
- 能看到商品页和后台各菜单

默认管理员入口：

- `/admin/login`

默认管理员账号请看根目录：

- [README.md](./README.md)

如果这里都还没通，先不要继续配业务。

## 第 2 步：后台每个页面是干什么的

### 1. 仪表盘

用途：

- 看整体运营概况
- 看订单、支付、履约、交付的大盘状态

建议：

- 日常巡检先看这里
- 这里只负责看，不负责精细配置

### 2. 商品中心

用途：

- 建商品
- 建价格模板
- 配可用支付方式
- 绑定履约策略和交付策略

你在这里最终决定的是：

- 用户买什么
- 卖多少钱
- 支持哪些支付方式
- 买完后走哪条履约和交付链路

### 3. 订单中心

用途：

- 查订单
- 看订单详情
- 做订单运营动作

常见动作：

- 确认付款
- 驳回付款
- 触发履约
- 重试履约
- 触发交付
- 补发
- 标记退款
- 发起原路退款

如果你每天只处理业务单据，这个页面会是最常用的页面之一。

### 4. 支付中心

用途：

- 看支付记录
- 看支付凭证
- 看支付回调日志
- 看链上扫单记录

什么时候用：

- 用户说“我已经付了”
- 用户上传了付款截图
- OKX 扫单异常
- 需要排查为什么订单没自动推进

### 5. 履约中心

用途：

- 看履约记录
- 看发码记录
- 看交付记录
- 重试失败任务

什么时候用：

- 支付已经确认，但商品还没真正发出去
- 上游接口失败
- 发码成功但交付失败

### 6. Telegram 中心

用途：

- 管理 Bot
- 管理 Webhook
- 看绑定关系
- 做 Telegram 测试发送

什么时候必须配：

- 你要用 Telegram 作为通知渠道
- 你要把交付结果发到 Telegram

### 7. 客户中心

用途：

- 看用户和客户相关信息
- 看购买行为和关联记录

当前系统前台主流程是匿名下单，所以这个页面更多偏运营视角，而不是前台必须依赖的账号体系。

### 8. 工单中心

用途：

- 看用户售后工单
- 分配工单
- 跟进处理

前台用户是通过 `订单号 + 查单码` 提交售后，不要求必须登录。

### 9. 系统中心

用途：

- 配支付通道
- 配 Provider / Action
- 配履约策略
- 配交付策略
- 配 Telegram 机器人
- 配内部客户端
- 配运行参数

如果说商品中心是“业务展示层”，那系统中心就是“业务底座层”。

## 第 3 步：怎么把一条商品卖起来

下面是最实用的一套流程。

### A. 配支付通道

去后台：

`系统中心 -> 支付通道`

当前主推荐保留这三种：

- `okx_usdt`
- `wechat_qr`
- `alipay_qr`

建议用法：

- `okx_usdt` 作为主支付
- `wechat_qr` / `alipay_qr` 作为人工兜底支付

如果微信、支付宝还没接官方支付接口：

- 仍然可以正常使用
- 直接走人工收款码 + 用户上传凭证 + 后台审核

### B. 配 Provider

去后台：

`系统中心 -> 集成服务`

这里是配置你的上游服务，例如：

- `new-api`
- 自己的发码服务
- 其他中间网关

先确定：

- 请求发到哪里
- 用什么鉴权
- 超时时间和重试次数

### C. 配 Action

去后台：

`系统中心 -> 集成动作`

这里配置的是：

- 对上游发哪个接口
- 用什么 HTTP 方法
- Body 长什么样
- 从返回里提取哪个字段

可以把它理解成：

- Provider 是“服务方”
- Action 是“调用服务方的哪个具体接口”

### D. 配履约策略

去后台：

`系统中心 -> 履约策略`

这里决定：

- 支付成功后怎么触发上游发码
- 从上游返回里拿什么结果
- 失败后怎么重试

### E. 配交付策略

去后台：

`系统中心 -> 交付策略`

这里决定：

- 发码结果通过什么方式发给用户
- 是站内显示
- 还是 Telegram 推送
- 还是人工交付

### F. 建商品

去后台：

`商品中心`

至少配置这些：

- 商品名称
- 商品类型
- 价格模板
- 支持的支付方式
- `fulfillment_strategy_key`
- `delivery_strategy_key`

商品真正能卖，不是因为“名字和价格填了”，而是因为这两个策略已经绑定好。

## 第 4 步：用户实际怎么买

当前前台主流程如下。

### 1. 用户进入商品页

用户可以直接浏览商品，不需要登录。

### 2. 用户选择支付方式并创建订单

系统创建订单后，会返回：

- `订单号`
- `查单码`
- 支付指引
- 收款码或收款内容

这里最重要的是：

- 用户必须保存 `订单号`
- 用户必须保存 `查单码`

因为后续所有订单操作都依赖这两个值。

### 3. 用户付款

分两种情况：

#### 人工收款

适用于：

- 微信二维码
- 支付宝二维码
- USDT 人工码

流程：

1. 用户扫码付款
2. 用户上传支付凭证
3. 后台审核
4. 订单推进

#### 自动收款

适用于：

- `okx_usdt`

流程：

1. 用户付款
2. 系统扫单或回调确认
3. 自动推进到已支付
4. 自动或人工进入履约

### 4. 用户后续怎么查订单

用户去前台订单查询页，用：

- `订单号`
- `查单码`

可以做这些事：

- 查订单状态
- 看交付结果
- 上传支付凭证
- 提交售后工单

不依赖前台账号体系。

## 第 5 步：运营实际怎么处理订单

### 场景 1：微信 / 支付宝人工审核

典型流程：

1. 用户下单
2. 用户支付
3. 用户上传截图
4. 后台去 `支付中心` 看凭证
5. 后台去 `订单中心` 确认付款
6. 系统进入履约
7. 履约成功后进入交付

### 场景 2：OKX 自动确认

典型流程：

1. 用户下单
2. 用户链上付款
3. `okx_usdt` 扫单任务确认到账
4. 系统自动把订单推进到已支付
5. 如果通道开启自动履约，则直接履约
6. 如果履约成功，继续交付

### 场景 3：履约失败

典型处理方式：

1. 去 `履约中心` 看失败记录
2. 看 Provider / Action 返回错误
3. 修正策略或上游参数
4. 重试履约

### 场景 4：交付失败

典型处理方式：

1. 去 `履约中心` 或 `订单中心` 看交付记录
2. 如果是 Telegram 失败，检查 Bot / Webhook / chat 绑定
3. 如果是站内失败，检查交付模板和结果字段
4. 必要时补发

### 场景 5：退款

当前后台有两种退款动作：

#### 标记退款

适用于：

- 你已经在线下或外部系统完成退款
- 只需要把 PassDock 内部状态改成已退款

效果：

- 立即把订单和支付改为 `refunded`
- 不调用外部支付接口

#### 发起原路退款

适用于：

- 你已经在支付通道里配好了退款 Provider / Action
- 需要系统直接调用微信 / 支付宝 / OKX / 其他渠道退款接口

效果：

- 记录退款状态
- 记录回执号
- 失败后可再次发起

如果退款接口没接通：

- 就不要点“发起原路退款”
- 只用“标记退款”

## 第 6 步：每天应该怎么巡检

建议每天至少看这几处。

### 开始工作前

先看：

1. 仪表盘
2. 订单中心
3. 支付中心
4. 履约中心

重点关注：

- 有没有待审核支付
- 有没有失败履约
- 有没有失败交付
- 有没有异常工单

### 处理中

优先顺序建议：

1. 先处理待审核支付
2. 再处理履约失败
3. 再处理交付失败
4. 最后处理退款和工单

原因：

- 钱先确认，订单才能继续流转
- 履约比售后更影响交付时效

### 收工前

检查：

- 有没有长时间 `pending_review` 的支付
- 有没有长时间 `processing` 的退款
- 有没有长时间未交付的已支付订单
- 有没有无人处理的高优先级工单

## 第 7 步：最推荐的上线方式

如果你现在的主业务是：

- `OKX` 主支付
- 微信 / 支付宝作为辅助
- 商品为卡密 / 订阅 / 激活码
- 交付希望兼容站内和 Telegram

推荐上线方式：

1. 先用 `wechat_qr` / `alipay_qr` 跑通人工审核闭环
2. 再加 `okx_usdt` 自动扫单
3. 再接真实 `new-api`
4. 再接 Telegram Bot

不要一开始就同时打通所有真实外部服务。

更稳妥的顺序是：

`人工支付闭环 -> 自动确认 -> 自动履约 -> Telegram 交付`

## 第 8 步：最常见的问题

### 1. 商品能看到，但下单后没发货

优先检查：

- 商品是否绑定了履约策略
- 履约策略是否绑定了可用的 Provider / Action
- 支付是否真的被确认

### 2. 用户说查不到订单

先确认用户有没有同时提供：

- `订单号`
- `查单码`

现在前台不是纯订单号查询。

### 3. 用户说我付了款，但系统没反应

看这几处：

- 支付中心是否有凭证
- 回调日志是否有记录
- 扫单记录是否有结果
- 订单状态是否还在 `awaiting_payment` 或 `pending_review`

### 4. “发起原路退款”按钮点不了

通常是因为：

- 当前订单状态不允许
- 支付通道没有配置退款 Provider / Action

### 5. Telegram 没收到交付

看这几处：

- Bot 是否启用
- Webhook 是否正确
- 用户是否已绑定
- 交付策略是否真的走 Telegram

## 最后的建议

如果你是系统管理员，最重要的不是一次把所有功能都打开，而是先确保这 4 条线真的通：

1. 能下单
2. 能确认支付
3. 能履约
4. 能交付

只要这四条通了，这个系统就已经具备真正可运营的价值。

后面的 Telegram、自动退款、监控、对象存储、上游多 Provider 扩展，都是在这个基础上继续增强。
